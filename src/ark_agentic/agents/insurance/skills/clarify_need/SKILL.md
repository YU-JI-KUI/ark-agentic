# 需求澄清技能

当用户的需求不够明确时，使用此技能通过友好的对话收集**缺失的**信息。

## 核心原则：不问已知信息

在提问之前，必须先检查用户已经说了什么。以下信息如果用户已经提供，**绝对不要重复询问**：

| 信息项 | 用户可能的表述 | 已知则跳过 |
|-------|-------------|-----------|
| 金额 | "取5万"、"需要10万"、"3万块" | 不再问金额 |
| 用途 | "交学费"、"看病"、"急用" | 不再问用途 |
| 紧急程度 | "很急"、"不着急"、"越快越好" | 不再问时间 |
| 保单选择 | "从年金险里取"、"用 POL001" | 不再问哪份保单 |

**只问用户没有提到的关键信息。如果所有关键信息都已知，直接进入方案计算，不要多此一举。**

## 触发条件

以下情况需要进行需求澄清：
- 用户说"我想取钱"但未说明金额、用途、紧急程度中的任何一项
- 用户需求可能有多种理解方式

注意：如果用户已经明确了金额（如"我想取5万"），本技能**不应触发**，应直接进入取款流程。

## 澄清原则

### 1. 最小化提问
- 每次只问 1-2 个**尚未明确的**关键问题
- 能从上下文推断的就不要问
- 非必要信息（如用途）可跳过

### 2. 自然衔接
- 承接用户已经提供的信息，不要忽略
- 例如用户说"取5万交学费"，已知金额和用途，只需确认紧急程度（或直接跳过，进入方案计算）

### 3. 友好语气
- 使用口语化表达
- 简短说明为什么需要这个信息
- 表达理解和耐心

## 常见澄清场景

### 只缺金额

用户说了"想取钱"但没说金额：

```
请问您大概需要多少资金呢？
```

不要给选项列表，简单一句话即可。

### 只缺紧急程度

用户说了金额但没说时间要求：

```
对到账时间有要求吗？急用的话有更快的方式。
```

### 只缺用途（可选信息）

用途不是必填项。如果金额和紧急程度已知，可以直接跳过用途，进入方案计算。只有在用途确实会影响推荐（如紧急医疗 vs 长期储蓄）时才问：

```
方便说一下用途吗？这样我能帮您找到更合适的方案。不方便说也没关系。
```

## 注意事项

1. **最重要**：不要重复询问用户已经明确的信息，这会显得很傻
2. 不要让用户觉得在被审问
3. 尊重用户隐私，非必要信息可跳过
4. 如果用户不愿回答，直接用已有信息给出通用方案
5. 不要在澄清中提及具体金额数字（如"可贷款3.3万"），金额需通过工具计算后才能给出
