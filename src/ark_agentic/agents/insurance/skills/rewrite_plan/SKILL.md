# 方案改写技能

当用户对推荐的方案提出修改意见时，使用此技能调整方案。

## 触发条件

用户表达以下意图时触发：
- "金额能多一点吗"
- "有没有更快到账的方案"
- "能不能不要影响保障"
- "不要贷款 / 不要退保"
- "换一个方案"
- "我觉得利息太高了"
- "POL002 少取一点"

## 第一步：判断调整类型

用户的修改请求分为三类，**必须先判断属于哪一类**，再选择对应的处理方式：

### 类型 A：调整总金额

用户想改变取款的总目标金额，但没有指定某张保单或某种方式。

典型表述：
- "我想多取一点，总共8万"
- "5万太多了，3万就够了"
- "能不能凑到10万"

**处理方式**：重新调用 `compare_plans`，用新金额重新排列方案。

```
rule_engine(action="compare_plans", user_id=用户ID, amount=新总金额)
```

然后按优先级展示新的方案列表，与之前的方案做对比。

### 类型 B：改变方案方向 / 排除某类方案

用户不改金额，但对方案类型有偏好或排除条件。

典型表述：
- "不要贷款"
- "不想退保"
- "只用不影响保障的方式"
- "有没有不收手续费的方案"
- "我不想动那个终身寿险"

**处理方式**：重新调用 `compare_plans` 获取完整方案列表，然后根据用户的约束条件**过滤展示**。

```
rule_engine(action="compare_plans", user_id=用户ID, amount=原金额)
```

拿到结果后：
- 排除用户不想要的方案类型（如排除 plan_type="policy_loan" 或 plan_type="surrender"）
- 只展示符合条件的方案
- 如果过滤后没有方案能满足金额，明确告知并给出替代建议

### 类型 C：调整某张保单 / 某个方案的具体金额

用户针对已推荐方案中的某一个做微调。

典型表述：
- "POL002 少取一点，取3万就行"
- "保单贷款能不能只贷2万"
- "红利那个能全取吗"

**处理方式**：用 `calculate_detail` 对目标保单做精确计算。

```
rule_engine(
  action="calculate_detail",
  policy={...从上文 compare_plans 结果中获取该保单数据...},
  plan_type="对应方案类型",
  amount=新金额
)
```

### 混合场景

用户可能同时提出多个条件，如 "多取一点但不要贷款"。处理方式：
1. 先用类型 A 的方式重新 `compare_plans`（新金额）
2. 再按类型 B 过滤掉不需要的方案
3. 展示过滤后的结果

## 第二步：生成调整方案

基于计算结果，向用户呈现：
1. **关联保单信息**：保单名称 + 保单号
2. **做了哪些调整**：简要说明变化
3. **新方案详情**：金额、费用、到账时间、保障影响
4. **与原方案对比**：费用差异、金额差异等
5. **建议与风险提示**

如果修改不可行（如过滤后无可用方案，或金额超出所有方案上限），清楚说明原因并提供替代建议。

## 第三步：操作确认

调整方案展示后，**必须**引导用户确认：
- 明确询问用户是否满意调整后的方案
- 用户确认后，告知可以"一键办理"
- 用户不满意则继续调整或提供其他替代方案

## 手续费参考

部分领取手续费率（按保单年度）：

| 保单年度 | 手续费率 |
|---------|---------|
| 第 1 年 | 3% |
| 第 2 年 | 2% |
| 第 3-5 年 | 1% |
| 第 6 年起 | 0% |

保单贷款：年利率 5%（固定），按日计息。

生存金领取、红利领取、退保：无手续费。

## 常见改写场景速查

| 用户说 | 调整类型 | 用哪个 action |
|-------|---------|--------------|
| "多取一点" / "总共要X万" | A 调总额 | `compare_plans`（新amount） |
| "不要贷款" / "不退保" | B 改方向 | `compare_plans` + 过滤 |
| "只用不影响保障的" | B 改方向 | `compare_plans` + 过滤 |
| "POL002 少取点" | C 调单保单 | `calculate_detail` |
| "贷款只贷2万" | C 调单保单 | `calculate_detail` |
| "多取一点但不要贷款" | A+B 混合 | `compare_plans`（新amount）+ 过滤 |
| "换个方案" | B 改方向 | `compare_plans` |
| "利息太高了" | B 改方向 | `compare_plans` + 排除贷款 |

## 输出格式

### 类型 A / B（重新推荐整套方案）

```markdown
### 调整后的方案

根据您的要求（[简述调整内容]），我重新为您计算了方案：

#### 方案一：[方案名称] ⭐ 推荐
- 📋 **关联保单**：[保单名称]（[保单号]）
- 💰 **可用金额**：[根据实际计算] 元
- 💵 **费用**：[根据实际计算]
- ⏱️ **到账时间**：1-3个工作日
- 🛡️ **对保障影响**：[具体说明]

#### 方案二：...

与之前相比：[简要对比]

请问这个方案可以吗？确认后我可以帮您一键办理。
```

### 类型 C（调整单个方案）

```markdown
### 调整后的方案

根据您的要求，我重新计算了 [保单名称] 的方案：

**[方案名称]**
- 📋 **关联保单**：[保单名称]（[保单号]）
- 💰 **金额**：[根据实际计算] 元（原方案：[原金额] 元）
- 💵 **费用**：[根据实际计算]（原方案：[原费用]）
- ⏱️ **到账时间**：1-3个工作日
- 🛡️ **对保障影响**：[具体说明]

确认后我可以帮您一键办理。如果还需要调整，请随时告诉我。
```

## 风格要求

- 友好、专业、简洁、通俗
- 避免机械表达，体现对用户需求的理解
- 金额使用千分位格式（如 65,000 元）
- 对比原方案时用简洁的方式呈现差异
- 不要自行编造金额，所有数字必须来自规则引擎计算

## 注意事项

1. 始终尊重用户的偏好
2. 必须先判断调整类型（A/B/C），再选择正确的工具 action
3. 每个方案必须标注关联保单的名称和保单号
4. 不可行时主动提供替代建议，说明原因
5. 方案展示后必须引导用户确认，不要直接结束对话
6. 退保方案的调整需要额外风险提示
