# 保险取款技能

当用户表达取款需求**并已明确金额**时，使用此技能帮助用户完成保险取款操作。

## 触发条件

**同时满足以下两个条件**时触发：
1. 用户表达了取款意图（"取钱"、"领取"、"借款"、"需要用钱"等）
2. 用户已明确了取款金额（如"取5万"、"需要10万"）

**不触发**的情况：
- 用户只说了"想取钱"但没说金额 → 应由 clarify_need 技能先收集金额
- 用户在聊非取款话题

## 执行流程

### 第一步：信息收集

调用 `customer_info` 获取用户基本信息（年龄、性别、家庭关系），用于后续推荐话术的针对性调整：

```
customer_info(info_type="identity", user_id=用户ID)
```

注意：**不需要**单独调用 `policy_query` 获取保单列表，规则引擎会在计算时自动获取。

### 第二步：调用规则引擎

调用规则引擎，传入 `user_id` 和金额：

```
rule_engine(
  action="list_options",
  user_id=用户ID,
  amount=用户期望金额
)
```

**不需要再做需求澄清**：金额在触发本技能前已由 clarify_need 技能确认。如果用途和紧急程度已知，可在话术中做调整；如果未知，按默认优先级推荐即可。

### 第三步：理解返回数据

规则引擎返回：

```json
{
  "requested_amount": 50000,
  "total_available_excl_loan": 45000,
  "total_available_incl_loan": 78000,
  "combination_hint": null 或 "提示文案",
  "options": [ ... ]
}
```

`options` 是按优先级排序的**可选项列表**。每个可选项代表一张保单的一个可用取款渠道（如生存金、红利、贷款等），包含该渠道的最大可用额度和费用信息。你需要从中组装最终推荐方案。

每个可选项的关键字段：
- `option_type` — 渠道类型（survival_fund / bonus / partial_withdrawal / surrender / policy_loan）
- `option_name` — 渠道中文名
- `available_amount` — **该渠道的最大可用额度（硬上限，不可超过）**
- `net_amount` — 按最大额度计算的扣费后净额
- `policy_id` / `product_name` — 保单信息
- `priority` / `rank` — 优先级和排名
- `coverage_impact` — 对保障的影响

顶层字段：
- `combination_hint` 为 null → 至少有一个可选项的 `net_amount` >= 用户期望金额
- `combination_hint` 不为 null → 没有单个可选项能满足，需组合或总额不足

### 第四步：组装推荐方案

基于可选项数据，组装出**一个推荐方案**（可能是单个可选项，也可能是多个可选项的组合）：

#### 情况一：有单个可选项能满足（combination_hint 为 null）

选择排名最高且 `net_amount` >= 用户期望金额的可选项，作为推荐方案。推荐金额 = 用户期望金额（不是 `available_amount`）。

#### 情况二：需要组合（combination_hint 不为 null，但总额够）

按优先级从高到低选取可选项，每个可选项取用金额**不超过其 `available_amount`**，直到累计满足用户期望金额：

1. 先用 P1 可选项（生存金 + 红利）
2. 不够再加 P2 可选项（部分领取）
3. 仍不够再加保单贷款
4. 退保作为最后手段

#### 情况三：所有可选项合计仍不足

`total_available_incl_loan` < 用户期望金额时，给出**最大可取方案**：
- 将所有可选项按优先级组合，每个取其最大可用额度
- 明确告知用户最多可取多少，询问是否接受或调整需求

### 第五步：方案呈现

**核心原则：永远只给一个推荐（一个 ⭐），加上 1-2 个备选。**

- **推荐方案**（1个）：标 ⭐ 推荐，详细展示
- **备选方案**（1-2个）：标"备选"，简要展示
- 推荐和备选可以是单个可选项或组合
- **每个可选项的取用金额不得超过其 `available_amount`**

**优先级参考**：
- P1（零成本）：生存金/满期金领取、红利领取 — 无手续费，不影响保障
- P2（低成本）：万能险/年金部分领取 — 手续费按年度（第1年3%、第2年2%、第3-5年1%、第6年起0%）
- P3（谨慎）：终身寿险退保 — 无手续费，但保障终止
- 特殊：保单贷款 — 年利率5%，适合紧急周转

### 第六步：操作确认

展示完方案后，**必须**引导用户确认选择：
- 询问用户希望选择哪个方案
- 用户确认后，告知可以"一键办理"
- 如果用户犹豫，可以进一步解释或调整

### 第七步：用户画像微调

根据 customer_info 返回的信息，在推荐**话术**中做针对性调整（注意：这是措辞调整，不改变方案排序）：

| 用户特征 | 推荐调整 |
|---------|---------|
| 年龄 > 55 岁 | 强调操作简便、到账快、对养老保障的影响最小 |
| 有子女（受益人含"子女"关系） | 提醒保留保障的重要性，避免退保影响家庭保障 |
| 资金用途=紧急 | 优先推荐保单贷款（保障不受影响，到账快） |
| 资金用途=教育/医疗 | 优先推荐零成本可选项（生存金+红利），如不够再考虑贷款 |

## 时效参考

| 方式 | 到账时间 | 手续费 | 对保障影响 |
|-----|---------|-------|-----------|
| 生存金领取 | 1-3个工作日 | 0 | 不影响 |
| 红利领取 | 1-3个工作日 | 0 | 不影响 |
| 部分领取 | 1-3个工作日 | 按年度费率 | 现金价值、保额同步下降 |
| 保单贷款 | 1-3个工作日 | 年利率5% | 正常不影响；逾期未还可致保单中止 |
| 退保 | 1-3个工作日 | 0 | 保障完全终止 |

## 输出格式

### 推荐为单个可选项时

```markdown
## 为您推荐的取款方案

### ⭐ 推荐：[option_name]
- 📋 **关联保单**：[product_name]（[policy_id]）
- 💰 **取款金额**：[用户期望金额] 元
- ⏱️ **到账时间**：1-3个工作日
- 💵 **费用**：无 / 手续费约[X]元
- 🛡️ **对保障影响**：[coverage_impact]
- 💡 **推荐理由**：[简短说明]

### 备选一：[方案名称]
- 📋 **关联保单**：...
- ...

请问您倾向于哪个方案？确认后我可以帮您一键办理。
```

### 推荐为组合方案时

```markdown
## 为您推荐的取款方案

由于单个渠道无法满足 [X] 元的需求，为您推荐以下组合方案：

### ⭐ 推荐：组合方案

| 步骤 | 方式 | 关联保单 | 金额 | 费用 | 保障影响 |
|-----|------|---------|------|------|---------|
| 1 | [option_name] | [product_name]（[policy_id]） | [≤available_amount] 元 | 无 | 不影响 |
| 2 | [option_name] | [product_name]（[policy_id]） | [≤available_amount] 元 | [X] 元 | [说明] |
| **合计** | | | **[总额] 元** | **[总费用] 元** | |

到账时间：均为 1-3 个工作日

### 备选一：[方案名称]
- ...

确认推荐方案后，我可以帮您一键办理。如果需要调整某一项，请告诉我。
```

### 总额不足时

```markdown
## 取款方案说明

经计算，您所有保单的可用渠道合计最多可取 **[total_available_incl_loan] 元**，无法完全满足 [requested_amount] 元的需求。

以下是最大可取方案：
...（同组合方案格式）

您可以：
1. 接受 [最大总额] 元的方案
2. 调整目标金额

请问您怎么看？
```

## 风格要求

- 友好、专业、简洁、通俗
- 避免机械表达和堆砌术语
- 金额使用千分位格式（如 65,000 元）
- 对敏感操作（退保、大额贷款）给出清晰风险提示
- 客观中立，不过度推销或劝退任何方案

## 注意事项

1. 始终优先推荐零成本、不影响保障的可选项
2. 每个方案必须标注关联保单的名称和保单号
3. 退保可选项仅在用户明确要求或确实无其他选择时才使用
4. **永远只有一个 ⭐ 推荐**：无论是单个可选项还是组合方案，只有一个标 ⭐
5. 方案展示后必须引导用户确认，不要直接结束对话
6. 遇到复杂情况或用户犹豫不决时，建议转人工服务
7. 所有金额必须来自规则引擎计算，不要自行编造数字
8. **不要做重复的需求澄清**：金额已在触发前由 clarify_need 技能确认，不要再问一遍
9. **金额硬约束**：任何可选项的取用金额绝不能超过其 `available_amount`
