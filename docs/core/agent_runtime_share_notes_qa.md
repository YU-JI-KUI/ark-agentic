## 一、开场部分

### 演讲重点
- 不要一开始就讲细节，先把项目定位讲清楚。
- 最好先说明：这不是一个普通的 LLM 调用封装。
- 重点强调它把 Agent 运行所需的关键能力都纳入了统一框架。

### 建议表达
“这个项目表面上是一个轻量级 ReAct Agent 框架，但真正值得讲的是，它把上下文工程、工具调用、技能系统、记忆管理、流式输出和卡片化展示都统一进了同一个运行时中。”

### 可能被问的问题

#### Q1：为什么要强调它不是普通的 LLM 封装？
**回答重点：**
- 普通封装通常只关注模型请求和响应
- 这里关注的是会话、技能、工具、记忆、输出协议等完整运行过程
- 从 `AgentRunner` 的实现看，它是一个循环执行器，而不是一次性调用器

#### Q2：ReAct 在这里体现在哪里？
**回答重点：**
- 体现在 `LLM → Tool → LLM` 的循环
- `AgentRunner._run_loop()` 负责这一流程
- 工具结果会进入下一轮推理上下文

## 二、整体架构部分

### 演讲重点
- 讲架构时不要逐目录念文件名，要突出“分层思想”。
- 重点说明哪些是框架共性层，哪些是业务装配层。
- 听众最关心的是模块边界是否清晰。

### 建议表达
“这套系统可以拆成五层：接入层负责暴露 API 和 CLI；装配层负责构造业务 Agent；Core Runtime 负责统一执行机制；领域适配层负责业务能力适配；资源层则承载技能、模板和 mock 数据。”

### 可能被问的问题

#### Q1：为什么不把所有逻辑都塞到 AgentRunner 里？
**回答重点：**
- 如果所有逻辑都塞进去，运行时会变成巨型 God Object
- 现在通过 `SessionManager`、`PromptBuilder`、`ToolRegistry`、`MemoryManager` 等组件拆分职责，更便于维护和替换

#### Q2：业务 Agent 和 Core Runtime 的边界是什么？
**回答重点：**
- Core 负责共性机制：调度、上下文、流式、记忆等
- 业务 Agent 负责注册自己的工具、技能、上下文预处理器
- 业务差异主要体现在 agent 装配与工具/模板上

## 三、AgentRunner 主执行链路

### 演讲重点
- 这一部分是全场的主线，建议讲清楚“输入进来后发生了什么”。
- 不必展开全部内部细节，但要把主循环讲顺。
- 重点强调“工具调用后的结果会继续反馈到后续推理”。

### 建议表达
“它不是把 prompt 拼好丢给模型就结束，而是先合并上下文，再构建系统提示，调用模型，如果模型决定用工具，就进入工具执行，再把结果回写到上下文中，继续下一轮推理。”

### 可能被问的问题

#### Q1：一轮执行结束的条件是什么？
**回答重点：**
- 当模型不再发起工具调用，或者达到某些流程结束条件时，生成最终结果
- 之后还会做输出校验，避免工具结果和回答不一致

#### Q2：工具执行是串行还是并行？
**回答重点：**
- 框架目标上支持并行工具调用
- 但是否并行与模型输出结构和 runner 调度策略相关
- 分享时可说这是 ReAct 框架比较关键的扩展能力之一

#### Q3：为什么说它更像运行时而不是 prompt chain？
**回答重点：**
- 因为它持有 session、tools、skills、memory、streaming 等多个运行态资源
- 模型调用只是运行时中的一个步骤

## 四、上下文工程部分

### 演讲重点
- 这是整个分享最重要的一段，建议多花一点时间。
- 重点不是“上下文很多”，而是“上下文被分层管理”。
- 强调上下文从 prompt 技巧升级成了系统能力。

### 建议表达
“在这个项目里，上下文不是一坨文本，而是多层来源：用户输入、session 状态、外部历史、技能、长期记忆、工具能力描述等。框架的核心价值就在于把这些来源统一编排。”

### 可能被问的问题

#### Q1：为什么说这是上下文工程，而不是普通 prompt engineering？
**回答重点：**
- prompt engineering 往往聚焦于写提示词
- 这里不仅有 prompt，还有状态、历史、技能、记忆、工具结果回写
- 它是多源上下文的系统治理问题

#### Q2：上下文多了，会不会导致冲突？
**回答重点：**
- 会，所以项目里有外部历史去重、context compaction、技能匹配等治理手段
- 这说明作者已经意识到上下文越多，治理成本越高

## 五、Session 与历史管理部分

### 演讲重点
- 重点讲两个点：短期上下文持久化，以及外部历史合并策略。
- 特别适合强调“工程化成熟度”。

### 建议表达
“它不是只在内存里存聊天记录，而是把 session、消息和状态都管理起来。更进一步，它还考虑了外部历史注入时如何避免重复和错位。”

### 可能被问的问题

#### Q1：为什么外部历史合并不能简单 append？
**回答重点：**
- 简单拼接会导致重复、顺序错乱、上下文污染
- 当前实现通过 pair-based fuzzy dedup 和 anchor insertion 让合并更稳

#### Q2：状态和消息为什么要分开？
**回答重点：**
- 消息记录对话过程
- 状态承载结构化上下文，如用户信息、工具结果、跨轮变量
- 两者职责不同，混在一起会增加 prompt 和工具编排复杂度

## 六、技能系统部分

### 演讲重点
- 强调技能系统不是“提示词碎片管理”，而是业务知识外部化机制。
- 如果听众关注可维护性，这一段会很有共鸣。

### 建议表达
“技能系统本质上是在做 SOP 外置：把业务流程、规则、建议、注意事项，从代码中拆出来放到 Markdown 技能中，再由运行时决定注入哪些技能。”

### 可能被问的问题

#### Q1：技能和系统 prompt 有什么区别？
**回答重点：**
- system prompt 更偏全局约束
- skill 更偏特定场景下的业务知识或操作规程
- skill 可以按 query/context 动态选择

#### Q2：为什么 dynamic skill 模式有价值？
**回答重点：**
- 避免每轮把所有 skill 全量塞进 prompt
- 先注入元数据，再按需读取全文，更节省上下文窗口
- 更适合技能数量多、内容长的场景

#### Q3：semantic 模式现在成熟吗？
**回答重点：**
- 当前仓库更像是接口预留，真正完整实现还不算落地
- 这也说明架构有前瞻性，但仍存在演进空间

## 七、工具系统部分

### 演讲重点
- 工具系统重点讲“统一协议”和“上下文回写”。
- 让听众意识到工具不仅是 action，也是状态演进的一部分。

### 建议表达
“这个框架里的工具不是普通函数集合，而是带 schema、带统一返回结构、可纳入运行时调度的能力单元。更关键的是，工具结果还能回写 session state，继续参与后续推理。”

### 可能被问的问题

#### Q1：为什么要做 ToolRegistry？
**回答重点：**
- 统一工具发现、schema 导出和运行时注册
- 便于 Agent 装配时按业务注入工具集
- 便于模型稳定感知当前可调用能力

#### Q2：工具结果为什么要写回 state？
**回答重点：**
- 为了支持多工具链路，例如“查数据 -> 生成卡片”
- 为了避免重复调用前序工具
- 为了让后续轮推理获得结构化事实

## 八、记忆管理部分

### 演讲重点
- 重点强调“短期会话”和“长期记忆”的分层设计。
- 听众容易把记忆理解成向量库，这里要纠正。

### 建议表达
“这个项目中的记忆不是单一概念，而是两层：一层是当前 session 的短期连续性，一层是可被检索召回的长期知识和用户画像。”

### 可能被问的问题

#### Q1：为什么不直接把所有历史都放到向量库？
**回答重点：**
- 当前轮推理仍然需要强连续性的 session history
- 向量库更适合召回，不适合代替所有对话上下文
- 所以这套设计更像‘会话管理 + 知识检索’的结合

#### Q2：长期记忆最核心的能力是什么？
**回答重点：**
- 混合检索：语义 + 关键词
- 支持按用户隔离
- 支持增量同步和结构化记忆存储

#### Q3：这套记忆能不能替换成外部数据库？
**回答重点：**
- 从项目待办与抽象设计来看，后续是可以演进的
- 当前架构已经有一定防腐层思路

- 两者解决的问题不同

## 十、流式设计部分

### 演讲重点
- 重点讲“事件化”和“协议适配”，不要只说 SSE。
- 这是框架对接前端和企业接口的重要部分。

### 建议表达
“它的流式设计不只是不断输出 token，而是先把内部过程转成事件，再通过 formatter 适配为不同输出协议。这样前端或其他系统接入时不需要绑定单一格式。”

### 可能被问的问题

#### Q1：为什么要支持多种流式协议？
**回答重点：**
- 不同消费端协议要求不同
- 企业场景通常需要标准化信封和事件格式
- 做协议层适配可以降低耦合度

#### Q2：thinking/final 分离的价值是什么？
**回答重点：**
- 可以内部保留思考信息处理能力
- 但最终输出只暴露应展示的部分
- 对安全性、产品体验和一致性都有帮助

## 十一、卡片设计部分

### 演讲重点
- 重点强调“协议化 UI 输出”。
- 不要把它讲成一个普通前端模板系统。

### 建议表达
“这里的卡片设计不是随手返回一段 JSON 给前端，而是通过 A2UI 协议建模成受约束的渲染事件。这意味着 UI 输出已经变成 Agent 运行时的一部分。”

### 可能被问的问题

#### Q1：为什么 A2UI 需要强校验？
**回答重点：**
- 如果卡片输出不受约束，前后端集成会非常脆弱
- 强校验有助于稳定渲染与错误定位

#### Q2：`components` 和 `catalogId` 为什么二选一？
**回答重点：**
- 代表两种渲染路径：直接组件树 vs 模板/目录驱动渲染
- 这种约束有助于保持协议清晰

#### Q3：为什么卡片设计值得单独讲？
**回答重点：**
- 因为它说明这个框架不只是文本问答
- 它已经开始考虑结构化输出与人机交互界面

## 十四、现场问答速答清单

### 问：这套系统最像 LangChain、AutoGen 还是工作流编排？
**答法建议：**
- 它借鉴了 Agent 编排思想，但更适合描述为“面向业务的轻量级 Agent Runtime”
- 它不是单纯 workflow，也不是纯多智能体框架，重点在上下文和执行机制统一

### 问：这套设计最关键的抽象是什么？
**答法建议：**
- `AgentRunner` 是运行时总控
- `Session + Skill + Tool + Memory` 是四个最关键的上下文抽象

### 问：如果删掉一个模块，最不能删哪个？
**答法建议：**
- 从框架主线看，最不能删的是 `AgentRunner`
- 从差异化价值看，最不能删的是“上下文工程相关能力”，也就是 session/skills/memory/prompt builder 这一组

### 问：它和传统 RAG 项目最大的差异是什么？
**答法建议：**
- RAG 重点是“检索后喂给模型”
- 这里重点是“Agent 在运行时如何管理多源上下文并执行工具”
- Memory 只是其中一层，不是全部

### 问：它适合什么类型的业务？
**答法建议：**
- 适合需要多轮对话、工具调用、业务 SOP、结构化输出的场景
- 例如金融、客服、知识助手、复杂流程引导类业务

## 十六、最推荐记住的三句总结

### 总结 1
`ark-agentic` 最核心的能力，不是调用模型，而是编排上下文。

### 总结 2
它把技能、工具、记忆、流式输出和卡片 UI 都纳入了统一的 Agent Runtime。

### 总结 3
它已经不是一个 demo 式 Agent，而是一套具备产品化潜力的运行时框架。
