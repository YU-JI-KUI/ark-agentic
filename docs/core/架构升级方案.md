# ark-agentic 智能体框架架构升级方案

> **版本**: v1.0
> **日期**: 2026-02-27
> **状态**: 方案设计完成，待评审

## 一、现状分析总结

### 1.1 当前架构特征

**ark-agentic 核心能力**:
- **单一 ReAct 模式**: 纯 Reason→Act→Observe 循环,无规划能力
- **单智能体架构**: AgentRegistry 仅支持静态注册,无动态路由
- **动态技能加载**: 支持 full/dynamic/semantic 三种模式(semantic 未实现)
- **会话管理**: JSONL 持久化 + LLM 摘要压缩
- **向量记忆**: FAISS + 混合搜索(RRF)

**代码规模**: ~9,762 行核心代码(53 个 Python 文件)

### 1.2 参考架构洞察

| 维度 | nanobot (极简) | ark-agentic (当前) | openclaw (全功能) | Google ADK (工业标准) |
|------|---------------|-------------------|------------------|---------------------|
| **智能体模式** | 单 ReAct | 单 ReAct | 多智能体 + 事件驱动 | 单/多混合 + 工作流智能体 |
| **规划能力** | 无 | 无 | 无(纯 ReAct) | ReAct + Planning-First 可选 |
| **技能加载** | 全量注入 | 动态加载(ReadSkillTool) | 全量注入 + 自动安装 | L1/L2/L3 增量加载 |
| **子智能体** | SpawnTool(后台任务) | 无 | 无明确子智能体 | sub_agents 层级结构 |
| **会话管理** | 简单 JSONL | LLM 摘要压缩 | 多阶段压缩 + 写锁 | 可插拔 SessionService |
| **上下文复杂度** | 1,382 行 | 3,500+ 行 | 50K+ 行 | 中等(协议驱动) |

### 1.3 核心问题识别

**问题 1: 单智能体 vs 多智能体路由**
- 当前: AgentRegistry 仅支持静态注册(`_agents: dict[str, AgentRunner]`),无意图识别
- 缺失: 动态智能体选择、子智能体委托、跨智能体状态传递

**问题 2: ReAct vs Planning**
- 当前: 纯 ReAct 循环,复杂任务无法分解
- 缺失: 任务规划器、子任务分解、执行计划验证

**问题 3: 上下文管理复杂度**
- 当前: 单层 LLM 摘要压缩,token 估算基于字符数
- 缺失: 多层压缩策略、跨会话记忆整合、上下文窗口守卫

**问题 4: 功能缺失对比**

| 功能 | ark-agentic | openclaw | nanobot | Google ADK |
|------|-------------|----------|---------|------------|
| **工作流智能体** | ❌ | ❌ | ❌ | ✅ Sequential/Parallel/Loop |
| **子智能体委托** | ❌ | ❌ | ✅ SpawnTool | ✅ sub_agents |
| **Planning-First** | ❌ | ❌ | ❌ | ✅ 可选 |
| **语义技能匹配** | ⚠️ 未实现 | ❌ | ❌ | ✅ L1/L2/L3 |
| **会话写锁** | ❌ | ✅ | ❌ | ✅ |
| **多层压缩** | ❌ | ✅ | ❌ | ✅ |

## 二、架构升级方案(渐进式三阶段)

遵循"够用简洁,留好拓展"原则,采用**渐进式升级**而非一次性重构。

### 阶段 0: 核心能力补齐(立即实施)

**目标**: 修复现有功能缺失,不改变架构

#### 0.1 实现语义技能匹配

**当前状态**: `semantic_matcher.py` 存在但未实现,`SkillConfig.default_load_mode` 的 `semantic` 模式回退到 `dynamic`

**实现方案**:
```python
# src/ark_agentic/core/skills/semantic_matcher.py
class SemanticSkillMatcher:
    def __init__(self, loader: SkillLoader, embedder: Embeddings):
        self.loader = loader
        self.embedder = embedder
        self._skill_embeddings: dict[str, np.ndarray] = {}

    async def match_by_query(
        self,
        query: str,
        top_k: int = 3,
        threshold: float = 0.6
    ) -> list[tuple[SkillEntry, float]]:
        """基于查询语义匹配技能"""
        query_embedding = await self.embedder.embed_query(query)

        # 计算余弦相似度
        scores = []
        for skill_id, skill_emb in self._skill_embeddings.items():
            score = cosine_similarity(query_embedding, skill_emb)
            if score >= threshold:
                skill = self.loader.get_skill(skill_id)
                scores.append((skill, score))

        return sorted(scores, key=lambda x: x[1], reverse=True)[:top_k]
```

**集成点**: `AgentRunner._build_system_prompt()` 中根据用户输入动态选择技能

**复杂度**: +200 行,复用现有 `Embeddings` 组件

#### 0.2 添加会话并发写锁

**当前问题**: 多请求并发修改同一会话时可能导致消息丢失

**实现方案**:
```python
# src/ark_agentic/core/session.py
import asyncio
from contextlib import asynccontextmanager

class SessionManager:
    def __init__(self, ...):
        self._write_locks: dict[str, asyncio.Lock] = {}

    @asynccontextmanager
    async def acquire_write_lock(self, session_id: str):
        """获取会话写锁"""
        if session_id not in self._write_locks:
            self._write_locks[session_id] = asyncio.Lock()

        async with self._write_locks[session_id]:
            yield

    async def add_message(self, session_id: str, message: AgentMessage):
        async with self.acquire_write_lock(session_id):
            # 原有逻辑
            ...
```

**复杂度**: +50 行,无破坏性变更

### 阶段 1: 智能体路由增强(核心升级)

**目标**: 支持多智能体动态路由,保持单 ReAct 模式

#### 1.1 智能体路由架构

**设计原则**:
- **单智能体为主**: 默认行为不变,保持简单
- **多智能体可选**: 通过配置启用,按需路由
- **动态加载技能**: 技能作为智能体能力的动态扩展

**核心组件**:

```python
# src/ark_agentic/core/routing/router.py
from typing import Protocol

class AgentSelector(Protocol):
    """智能体选择器协议"""
    async def select(
        self,
        user_input: str,
        context: dict[str, Any]
    ) -> str:
        """返回智能体 ID"""
        ...

class RuleBasedSelector:
    """基于规则的选择器(阶段 1)"""
    def __init__(self, rules: list[tuple[str, Callable[[str], bool]]]):
        self.rules = rules  # [(agent_id, predicate)]

    async def select(self, user_input: str, context: dict) -> str:
        for agent_id, predicate in self.rules:
            if predicate(user_input):
                return agent_id
        return "default"  # 回退到默认智能体

class LLMBasedSelector:
    """基于 LLM 的选择器(阶段 2 可选)"""
    def __init__(self, llm: BaseChatModel, agent_registry: AgentRegistry):
        self.llm = llm
        self.registry = agent_registry

    async def select(self, user_input: str, context: dict) -> str:
        # 构建智能体描述
        agents_desc = "\n".join([
            f"- {aid}: {self.registry.get_description(aid)}"
            for aid in self.registry.list_ids()
        ])

        prompt = f"""根据用户输入选择最合适的智能体:
{agents_desc}

用户输入: {user_input}
返回智能体 ID:"""

        response = await self.llm.ainvoke(prompt)
        return response.content.strip()
```

**AgentRegistry 增强**:
```python
# src/ark_agentic/app.py
class AgentRegistry:
    def __init__(self, selector: AgentSelector | None = None):
        self._agents: dict[str, AgentRunner] = {}
        self._descriptions: dict[str, str] = {}
        self._selector = selector or RuleBasedSelector([])

    def register(
        self,
        agent_id: str,
        agent: AgentRunner,
        description: str = ""
    ):
        self._agents[agent_id] = agent
        self._descriptions[agent_id] = description

    async def route(
        self,
        user_input: str,
        context: dict[str, Any]
    ) -> AgentRunner:
        """动态路由到合适的智能体"""
        agent_id = await self._selector.select(user_input, context)
        return self.get(agent_id)
```

**API 层集成**:
```python
# src/ark_agentic/app.py
@app.post("/chat")
async def chat(request: ChatRequest):
    # 动态路由(如果未指定 agent_id)
    if not request.agent_id:
        agent = await _registry.route(request.message, request.context or {})
    else:
        agent = _get_agent(request.agent_id)

    # 原有执行逻辑
    ...
```

**复杂度**: +300 行,向后兼容(默认行为不变)

#### 1.2 技能路由策略对比

| 策略 | 实现方式 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **单智能体 + 动态技能** | 当前方案 | 简单,上下文统一 | 技能冲突风险 | 技能数量 <20,领域单一 |
| **多智能体 + 固定技能** | 每个智能体预配置技能 | 隔离性好,专业化 | 路由复杂度高 | 跨领域任务,技能 >50 |
| **混合模式(推荐)** | 默认智能体 + 专家智能体 | 平衡简单性和能力 | 需要路由逻辑 | 通用场景 |

**推荐方案**: 混合模式
- **默认智能体**: 加载通用技能(文件操作、搜索、记忆)
- **专家智能体**: 按需路由(保险理赔、金融分析、代码审查)
- **技能共享**: 通过 `SkillLoader` 的命名空间机制避免冲突

**配置示例**:
```python

    create_default_agent(skill_dirs=["skills/common"]),
    description="通用助手,处理文件、搜索、记忆等基础任务"
)

# 注册专家智能体
_registry.register(
    "insurance",
    create_insurance_agent(skill_dirs=["skills/insurance"]),
    description="保险理赔专家,处理保单查询、理赔审核、风险评估"
)

# 配置路由规则
selector = RuleBasedSelector([
    ("insurance", lambda q: any(kw in q for kw in ["保单", "理赔", "保险"])),
    ("default", lambda q: True)  # 回退
])
```

### 阶段 2: 规划能力增强(可选升级)

**目标**: 支持 Planning-First 模式,保持 ReAct 为默认

#### 2.1 规划器设计

**核心思想**: 参考 Google ADK,通过 `planner` 配置切换模式

```python
# src/ark_agentic/core/planning/planner.py
from typing import Protocol

class Planner(Protocol):
    """规划器协议"""
    async def plan(
        self,
        user_input: str,
        context: dict[str, Any]
    ) -> list[str]:
        """返回子任务列表"""
        ...

class ReActPlanner:
    """默认 ReAct 规划器(无规划)"""
    async def plan(self, user_input: str, context: dict) -> list[str]:
        return [user_input]  # 直接返回原始输入

class LLMPlanner:
    """基于 LLM 的规划器"""
    def __init__(self, llm: BaseChatModel):
        self.llm = llm

    async def plan(self, user_input: str, context: dict) -> list[str]:
        prompt = f"""将以下任务分解为可执行的子任务:

任务: {user_input}

要求:
1. 每个子任务应该是原子操作
2. 子任务之间有明确的依赖关系
3. 按执行顺序排列

返回 JSON 格式:
{{"subtasks": ["子任务1", "子任务2", ...]}}
"""
        response = await self.llm.ainvoke(prompt)
        data = json.loads(response.content)
        return data["subtasks"]
```

**AgentRunner 集成**:
```python
# src/ark_agentic/core/runner.py
class RunnerConfig:
    planner: Planner = field(default_factory=ReActPlanner)  # 新增

class AgentRunner:
    async def run(self, session_id: str, user_input: str, ...):
        # 规划阶段
        subtasks = await self.config.planner.plan(user_input, context)

        # 执行阶段
        results = []
        for subtask in subtasks:
            result = await self._run_loop(session_id, subtask, ...)
            results.append(result)

        # 聚合结果
        return self._aggregate_results(results)
```

**复杂度**: +250 行,向后兼容(默认 ReActPlanner)

---

#### 2.2 ReAct vs Planning 对比

| 维度 | ReAct(当前) | Planning-First(可选) |
|------|------------|---------------------|
| **适用场景** | 简单任务,单步操作 | 复杂任务,多步骤 |
| **上下文消耗** | 低(单轮对话) | 高(规划 + 执行) |
| **可控性** | 低(LLM 自主决策) | 高(明确执行计划) |
| **失败恢复** | 困难(无计划) | 容易(重新规划) |
| **实现复杂度** | 简单(已实现) | 中等(+250 行) |

**推荐策略**:
- **默认 ReAct**: 保持当前行为
- **可选 Planning**: 通过 `RunOptions.planner` 覆盖
- **自动切换**: 根据任务复杂度(如输入长度 >200 字)自动启用规划

### 阶段 3: 上下文管理优化(性能优化)

**目标**: 降低上下文消耗,提升长会话性能

#### 3.1 多层压缩策略

**当前问题**: 单层 LLM 摘要,所有历史消息一次性压缩

**优化方案**: 参考 openclaw 的多阶段压缩

```python
# src/ark_agentic/core/compaction.py
class MultiTierCompactor:
    """多层压缩器"""

    async def compact(
        self,
        messages: list[AgentMessage]
    ) -> CompactionResult:
        # 第 1 层: 近期消息(保留原始)
        recent = messages[-self.config.preserve_recent:]

        # 第 2 层: 中期消息(摘要压缩)
        mid_term = messages[-50:-self.config.preserve_recent]
        mid_summary = await self._summarize_chunk(mid_term)

        # 第 3 层: 远期消息(写入记忆)
        long_term = messages[:-50]
        await self._flush_to_memory(long_term)

        return CompactionResult(
            summary=mid_summary,
            preserved_messages=recent,
            removed_count=len(long_term)
        )
```

**复杂度**: +150 行,替换现有 `ContextCompactor`

---

#### 3.2 上下文窗口守卫

**当前问题**: Token 估算基于字符数,不准确

**优化方案**: 使用 tiktoken 精确计算

```python
# src/ark_agentic/core/compaction.py
import tiktoken

def estimate_message_tokens_precise(
    messages: list[AgentMessage],
    model: str = "gpt-4"
) -> int:
    """精确 token 估算"""
    encoding = tiktoken.encoding_for_model(model)
    total = 0
    for msg in messages:
        # 消息格式开销
        total += 4  # <|start|>role<|end|>
        total += len(encoding.encode(msg.content))
        if msg.tool_calls:
            for tc in msg.tool_calls:
                total += len(encoding.encode(json.dumps(tc.arguments)))
    return total
```

**复杂度**: +50 行,依赖 `tiktoken`

## 五、架构决策记录(ADR)

### ADR-001: 渐进式升级 vs 重构

**决策**: 采用渐进式升级

**理由**:
1. **风险控制**: 避免大规模重构引入的回归风险
2. **业务连续性**: 保持现有功能可用
3. **学习曲线**: 团队逐步适应新架构
4. **投资回报**: 优先实现高价值功能

**替代方案**: 完全重构(参考 openclaw 架构)
- 优点: 架构更清晰,扩展性更好
- 缺点: 风险高,周期长(3-6 个月),业务中断

---

### ADR-002: 单智能体 + 动态技能 vs 多智能体 + 固定技能

**决策**: 混合模式(默认智能体 + 专家智能体)

**理由**:
1. **简单性**: 默认智能体保持简单,满足 80% 场景
2. **专业性**: 专家智能体处理复杂领域任务
3. **可扩展性**: 通过路由规则灵活扩展
4. **上下文效率**: 专家智能体减少无关技能干扰

**替代方案**: 纯多智能体架构
- 优点: 隔离性好,专业化程度高
- 缺点: 路由复杂度高,上下文碎片化

---

### ADR-003: ReAct vs Planning-First

**决策**: ReAct 为默认,Planning 可选

**理由**:
1. **够用简洁**: ReAct 满足大部分场景
2. **性能**: Planning 增加上下文消耗
3. **向后兼容**: 保持现有行为
4. **灵活性**: 按需启用 Planning

**替代方案**: Planning 为默认
- 优点: 复杂任务处理能力强
- 缺点: 简单任务性能下降,上下文消耗高

---

## 六、总结与建议

### 6.1 核心原则

1. **够用简洁**: 优先实现高价值功能,避免过度设计
