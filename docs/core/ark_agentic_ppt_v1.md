## 第 2 页：背景——Agent 落地中的“最后一公里”痛点

### 页面目标
说明为什么需要这样一套框架：Demo 容易，生产化困难。

### 布局方式
**左右结构**
- 左侧 70%：痛点与问题本质
- 右侧 30%：框架对应解决方向

### 70% 核心内容
#### 核心标题
**Demo 很简单，但生产化很难**

#### 四个典型痛点
1. **逻辑碎片化**：业务 SOP 硬编码在 Prompt 中，难以维护、难以协作。
2. **上下文失控**：长对话容易堆积脏历史，缺乏状态管理、压缩和去重机制。
3. **UI 交付难**：流式输出、卡片、表单等复杂前端交互缺乏统一协议。
4. **架构过重**：有些主流框架更适合复杂编排，但对业务落地团队不够轻便。

#### 这一页想传达的判断
真正难的不是“让模型回答”，而是“让 Agent 作为系统稳定运行”。

### 30% 技术补充
#### Ark-Agentic 对应解决思路
- 用 Skills 解决 SOP 资产化
- 用 Session / History Merge / Compaction 治理上下文
- 用 AGUI / A2UI 解决输出协议和富交互问题
- 用轻量 ReAct Runtime 保持整体工程复杂度可控

#### 相关技术 ICON
- SOP / Playbook
- Context Management
- Compression
- AGUI Protocol
- Interactive Cards

### 70% 核心内容
#### 核心公式
**Agent = LLM + Tools + Skills + Session/State + Memory + Streaming/UI**

#### 公式拆解

- 避免把所有逻辑塞进单一 prompt

## 第 9 页：AgentRunner——整个系统的执行中枢

### 页面目标
讲清楚“输入进来之后到底发生了什么”。

### 布局方式
**左右结构**
- 左侧 70%：主执行链路
- 右侧 30%：关键方法与工程意义

### 70% 核心内容
#### 核心判断
**AgentRunner 不是一次性模型调用器，而是完整的 ReAct 循环执行器。**

#### 主执行链路
1. 读取 session 与历史消息
2. 合并 `input_context`、外部历史和 state
3. 必要时触发上下文压缩
4. 构建 system prompt
5. 调用 LLM
6. 解析 tool calls
7. 执行工具并写回结果
8. 驱动下一轮推理
9. 生成最终回答并进行结果校验

### 30% 技术补充
#### 关键方法
- `run()`：统一入口
- `_run_loop()`：ReAct 主循环
- `_build_system_prompt()`：上下文编排点
- `_call_llm_streaming()`：流式调用
- `_execute_tools()`：工具调度与结果回写

#### 相关技术 ICON
- Runner
- Loop
- Tool Call
- State Update

### 70% 核心内容
#### 核心判断
**Agent 不只是返回文本，而是在驱动结构化交互界面。**

#### 协议价值

#### 这页建议强调

### 30% 技术补充
#### A2UI 关键约束
- 顶层事件包括 `beginRendering / surfaceUpdate / dataModelUpdate / deleteSurface`
